== NAME

gitcommitizen - configuration file for conventional commit linting

== SYNOPSIS

*.gitcommitizen*

== DESCRIPTION

The **.gitcommitizen** file defines commit message types, scopes, and validation
settings for conventional commits. It uses INI-style configuration with three
sections: settings, scopes, and types.

== FILE FORMAT

INI-style configuration with three sections:

[.literal]
----
[settings]
require-scope = false
defined-scope = false
enforce-patterns = false
multi-scope = false

[scopes]
name = glob-pattern[, glob-pattern]...

[types]
name = description
----

=== Settings

*require-scope*::
  Require a scope to be present (default: false). When enabled:
  - *create*: Removes the "(none)" option; scope is mandatory.
  - *lint*: Fails if commit has no scope.
  Can be overridden by `-r` or `--no-require-scope` flags.

*defined-scope*::
  Require scope to be defined in configuration (default: false). When enabled:
  - *create*: Removes the "(custom)" option; only configured scopes allowed.
  - *lint*: Fails if scope is not in `[scopes]`.
  Can be overridden by `-d` or `--no-defined-scope` flags.

*enforce-patterns*::
  Enforce scope-file pattern matching (default: false). When enabled:
  - Requires scope to match file patterns from `[scopes]` when `--paths` is provided.
  - Implies `defined-scope = true` (scope must exist in config).
  Can be overridden by `-e` or `--no-enforce-patterns` flags.

*multi-scope*::
  Allow multiple scopes like `feat(api,db):` (default: false).
  Can be overridden by `-m` or `--no-multi-scope` flags.

=== Scopes

Maps scope names to glob patterns. Multiple patterns are separated by commas.
When validating with `--files`, the scope is checked against these patterns
to ensure commits use appropriate scopes for the files being changed.

Special scope `*` matches any files (wildcard). This allows any scope to be used
regardless of which files are staged.

=== Types

Maps type names to their descriptions. These are shown during interactive commit
creation and used for validation.

== SYNTAX

Lines starting with `#` or `;` are comments. Blank lines are ignored.
Whitespace around `=` is optional.

== EXAMPLES

[.literal]
----
[settings]
require-scope = true
defined-scope = true
enforce-patterns = true
multi-scope = true

[scopes]
# API and database scopes
api = src/api/**, tests/api/**
db = src/db/**, migrations/**
ui = src/components/**, src/pages/**
docs = docs/**, *.md
* = .github/**, scripts/**

[types]
feat = A new feature
fix = A bug fix
docs = Documentation only changes
style = Code style changes (formatting, semicolons)
refactor = Code refactoring without behavior change
perf = Performance improvements
test = Adding or updating tests
build = Build system or dependencies
ci = CI/CD configuration changes
chore = Maintenance tasks
revert = Revert a previous commit
----

This configuration:

* Requires a scope to be present (`require-scope = true`)
* Requires scope to be defined in config (`defined-scope = true`)
* Enforces scope-file pattern matching (`enforce-patterns = true`)
* Allows multi-scopes like `feat(api,db):` for cross-cutting changes
* Maps scopes to file patterns for path-based validation
* Defines standard commit types with descriptions

== FILE LOCATION

Place **.gitcommitizen** in the root of a Git repository.

== NOTES

* Section names (`[settings]`, `[scopes]`, `[types]`) are case-insensitive.
* Types are lowercase identifiers (e.g., `feat`, `fix`).
* Scopes may include letters, numbers, dashes, underscores, and `@` (e.g., `@types/node`).
* Glob patterns follow standard shell globbing with `**` for recursive matching.
