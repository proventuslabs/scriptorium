== NAME

env - environment variable file format

== SYNOPSIS

*.env* files contain environment variable definitions in key-value format.

== DESCRIPTION

The dotenv file format provides a simple method for defining environment
variables in plain text files. These files typically use the *.env* extension
and contain variable definitions that can be loaded into the environment of
applications or processes.

== FILE FORMAT

=== Basic Syntax

The basic syntax consists of key-value pairs separated by an equals sign:

----
KEY=value
ANOTHER_KEY=another value
----

=== Key Naming Rules

Variable names must follow these conventions:

* Must start with a letter (a-z, A-Z) or underscore (_)
* Can contain letters, digits (0-9), and underscores
* Case sensitive (typically uppercase by convention)
* Cannot contain spaces or special characters except underscore

Valid examples:

----
DATABASE_URL=example
_PRIVATE_KEY=secret
API_VERSION_2=v2.1
----

== QUOTING BEHAVIOR

The dotenv format supports three quoting modes with distinct behaviors:

=== Unquoted Values (Literal)

Unquoted values are treated literally with leading and trailing whitespace
trimmed:

----
NAME=John Doe
PATH=/usr/local/bin
DEBUG=true
----

Special characters, escape sequences, and variable references are treated as
literal text.

=== Single-Quoted Values (Literal with Minimal Escaping)

Single-quoted values preserve all characters literally. The only supported
escape sequence is *\'* for including a literal single quote:

----
MESSAGE='Hello World'
PATH_WITH_SPACES='  /path/with spaces  '
LITERAL_BACKSLASHES='C:\Windows\System32'
QUOTE_EXAMPLE='It'\''s a beautiful day'
JSON_CONFIG='{"key": "value", "number": 42}'
----

Within single quotes:

* Backslashes are literal (no escape processing)
* Variable substitution is disabled
* Newlines and special characters are preserved
* Only \' is interpreted as an escaped single quote

=== Double-Quoted Values (Full Processing)

Double-quoted values support full escape sequence processing and variable
substitution:

----
MESSAGE="Hello \"World\""
MULTILINE="Line 1\nLine 2\nLine 3"
PATH="C:\\Windows\\System32"
GREETING="Hello $USER, welcome!"
CONFIG_PATH="${HOME}/.config/myapp"
----

*Supported escape sequences:*

* \" - literal double quote
* \\ - literal backslash
* \n - newline character
* \t - tab character
* \r - carriage return
* \f - form feed
* \b - backspace
* \v - vertical tab

== MULTILINE VALUES

Multiline values are supported only within quoted strings (single or double
quotes):

=== Single-Quoted Multiline

----
PRIVATE_KEY='-----BEGIN PRIVATE KEY-----
MIIEvQIBADANBgkqhkiG9w0BAQEFAASCBKcwggSjAgEAAoIBAQC7
VGhT7sMV0f7KfFQk8vX9YgH2pL9eN3mK1jR5wS8uA7B6cD9fE0
-----END PRIVATE KEY-----'
----

=== Double-Quoted Multiline

----
SQL_QUERY="SELECT users.id, users.name, profiles.bio
FROM users
INNER JOIN profiles ON users.id = profiles.user_id
WHERE users.active = true"
----

== VARIABLE SUBSTITUTION

Variable substitution is performed only within double-quoted values. Three
forms are supported:

=== Basic Substitution ($VAR)

----
HOME=/home/user
CONFIG_PATH="$HOME/.config"
----

=== Explicit Boundaries (${VAR})

----
PREFIX=app
LOGFILE="${PREFIX}_error.log"
----

=== Default Values (${VAR:-default})

----
DATABASE_HOST="${DB_HOST:-localhost}"
PORT="${SERVER_PORT:-3000}"
DEBUG_MODE="${DEBUG:-false}"
----

*Substitution Rules:*

* Variables are resolved from the current environment
* If a variable is undefined, $VAR and ${VAR} expand to empty string
* ${VAR:-default} uses the default value if VAR is undefined or empty
* Nested substitution is not supported
* Substitution occurs only in double-quoted contexts

== COMMENTS

=== Full-Line Comments

Lines beginning with # (optionally preceded by whitespace) are comments:

----
# Application configuration
DATABASE_URL=postgresql://localhost:5432/mydb

    # Cache settings
REDIS_HOST=localhost
----

=== End-of-Line Comments

Comments can appear at the end of variable definitions:

----
DATABASE_URL=postgresql://localhost:5432/mydb  # Production database
DEBUG=false  # Disable debug mode in production
API_KEY=secret123  # TODO: rotate this key monthly
----

*Comment Rules:*

* Comments extend from # to the end of the line
* # characters within quoted strings are literal, not comment markers
* Leading whitespace before # is ignored for full-line comments

== BLANK LINES AND WHITESPACE

* Blank lines and lines containing only whitespace are ignored
* Leading and trailing whitespace around keys and unquoted values is trimmed
* Whitespace within quoted values is preserved exactly

== PROCESSING ORDER

Variables are processed in the order they appear in the file:

----
BASE_URL=https://api.example.com
API_ENDPOINT="$BASE_URL/v1/users"  # References BASE_URL defined above
----

Later definitions may reference earlier ones, but forward references are not
supported.

== EXAMPLES

=== Basic Configuration

----
# Database settings
DATABASE_HOST=localhost
DATABASE_PORT=5432
DATABASE_NAME=myapp

# Application settings
DEBUG=true
LOG_LEVEL=info
----

=== Complex Configuration with Substitution

----
# Environment setup
ENV=production
APP_NAME=myservice

# Paths
HOME_DIR=/opt/myservice
CONFIG_DIR="${HOME_DIR}/config"
LOG_DIR="${HOME_DIR}/logs"

# Database configuration
DB_HOST="${DATABASE_HOST:-localhost}"
DB_NAME="${APP_NAME}_${ENV}"
DATABASE_URL="postgresql://${DB_USER}:${DB_PASS}@${DB_HOST}:5432/${DB_NAME}"

# Multiline private key
PRIVATE_KEY="-----BEGIN RSA PRIVATE KEY-----
MIIEpAIBAAKCAQEA2Z9QJG8YgN7jkF1L3vX9YgH2pL9eN3mK1jR5
wS8uA7B6cD9fE0GhT7sMV0f7KfFQk8vX9YgH2pL9eN3mK1jR5wS
-----END RSA PRIVATE KEY-----"
----

=== Quoting Examples

----
# Unquoted - spaces trimmed
MESSAGE=Hello World

# Single quoted - literal preservation
JSON_DATA='{"status": "ok", "count": 42}'
WINDOWS_PATH='C:\Program Files\MyApp'

# Double quoted - full processing
FORMATTED_MESSAGE="Welcome ${USER}!\nToday is $(date)"
ESCAPED_QUOTES="He said \"Hello\" to me"
----

== COMPATIBILITY NOTES

This specification defines a standard dotenv format. Individual implementations
may vary in:

* Support for additional escape sequences
* Handling of malformed syntax
* Variable resolution precedence
* Maximum line length limits

Applications should document any deviations from this standard format.
