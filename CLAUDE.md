# Claude Context

Shell script collection with build system. Bash 4+.

See [CONTRIBUTING.md](CONTRIBUTING.md) for development commands and build system.

## Key Files

- `scripts/<name>/main.sh` - Script entry point
- `scripts/<name>/options.sh` - CLI options (getoptions DSL)
- `scripts/<name>/default.nix` - Nix package definition
- `scripts/<name>/docs/<name>.adoc` - Manpage (AsciiDoc)
- `scripts/<name>/completions/` - Shell completions (`.bash`, `_zsh`)
- `scripts/<name>/*_spec.sh` - Tests (ShellSpec)
- `templates/` - Script templates for `make new-script`
- `flake.nix` - Nix packages and dev environment
- `utils/bundle.sh` - Bundles scripts into single executables

## Conventions

- Inline shellcheck directives with an explanation when possible
- Use `#!/usr/bin/env bash` shebang
- Use `utils/bundle.sh` for build-time code generation
- Runtime getoptions eval, bundle-time gengetoptions parser generation

## CLI Flag Changes

When renaming or modifying CLI flags, update all locations:

- `options.sh` - flag definition and variable name
- `main.sh` / `cmd_*.sh` - variable usage
- `docs/*.adoc` - documentation
- `completions/*.bash` and `_*` - shell completions
- `*_spec.sh` - tests (verify exact flags, not just keywords)
- `.github/workflows/` - CI scripts using the command
- Internal commands generating scripts (e.g., hook templates)

## Implementation Plans

- Plan files (`docs/plans/*.md`) are local artifacts for implementation guidance
- Do NOT commit plan files - they are gitignored
- Create plans before complex multi-step implementations
- Delete or keep locally after implementation is complete

## Pull Requests

- Always run tests, lint, build using make targets before pushing
- Do not include Claude mentions, "Generated by Claude", or co-authoring attribution in commits or PRs
- PR descriptions can be detailed for reviewers (Summary, Changes, Breaking Change sections)
- Squash commit message should be concise following conventional commits
- Breaking changes use `!` after scope: `fix(scope)!: message`
